# AgentOps Hub 企业智能运营协同平台项目框架

本文档用于描述 `AgentOps Hub` 企业级 Agent AI 工程实践项目的需求分析、系统架构、推荐代码结构与数据库设计。项目目标是构建一个具备多 Agent 协同、前后端分离、Java + Python 双后端、Spring AI、LangChain、LangGraph、中间件、Kubernetes 部署和完整测试体系的企业级智能运营平台。

## 1. 需求分析

### 1.1 业务背景

企业在客户服务、运营分析、风险处置和知识管理中通常存在以下问题：

- 业务数据分散在工单系统、客户系统、知识库和 BI 系统中，人工分析成本高。
- 客服与运营人员需要反复查询历史工单、客户资料、SOP 文档和处理记录。
- 投诉增长、SLA 风险、客户流失风险等问题往往发现滞后。
- 传统 AI 问答只能回答问题，难以完成跨系统的业务动作。

`AgentOps Hub` 的核心目标是用多 Agent 工作流把企业内部数据、知识库、业务系统和人工审批连接起来，让 AI 从“回答问题”升级为“协同完成任务”。

### 1.2 项目核心目标

- 建设一个企业内部智能运营工作台，支持自然语言发起复杂业务任务。
- 构建 Java + Python 双后端架构，Java 负责企业业务系统，Python 负责 Agent 编排与 AI 能力。
- 使用 Spring AI 承接 Java 侧轻量 AI 能力和企业系统集成。
- 使用 LangChain 和 LangGraph 实现 RAG、多 Agent 协作、状态流转和人工确认节点。
- 支持消息队列、缓存、向量数据库、对象存储、搜索引擎等中间件。
- 支持 Docker、Kubernetes、Helm、监控、链路追踪和自动扩缩容。
- 覆盖单元测试、集成测试、系统测试、压力测试和 AI 质量评测。

### 1.3 目标用户与角色

| 角色 | 核心诉求 | 典型操作 |
|---|---|---|
| 运营人员 | 快速分析工单、投诉、客户反馈和业务异常 | 发起分析任务、查看报告、确认处置建议 |
| 客服主管 | 提升工单分派效率和 SLA 达成率 | 查看风险工单、审批自动派单、跟踪处理进度 |
| 知识库管理员 | 维护 FAQ、SOP、政策文档和案例库 | 上传文档、管理知识库、检查 RAG 引用质量 |
| 系统管理员 | 管理用户、权限、模型配置和系统监控 | 配置租户、角色权限、模型供应商、告警规则 |
| AI 工程师 | 优化 Agent 工作流、Prompt、工具调用和评测指标 | 调整 LangGraph 节点、评估 RAG 效果、分析链路追踪 |

### 1.4 典型业务流程

以“分析最近 7 天客户投诉上涨原因，找出高风险客户，生成处理建议，并创建跟进任务”为例：

1. 用户在前端工作台输入自然语言任务。
2. API Gateway 完成鉴权、限流和路由。
3. Java 后端创建 AI 任务记录，并将任务投递到消息队列。
4. Python Agent 服务消费任务，进入 LangGraph 工作流。
5. `Planner Agent` 解析用户意图并拆解执行步骤。
6. `Retriever Agent` 检索知识库、历史工单和客户资料。
7. `Data Analyst Agent` 分析投诉趋势、分类分布、异常指标和热点问题。
8. `Risk Agent` 识别高风险客户、潜在 SLA 风险和异常工单。
9. `Supervisor Agent` 校验结论完整性、引用可靠性和动作风险。
10. 如果涉及高风险动作，系统进入人工审批节点。
11. 审批通过后，`Action Agent` 调用 Java 业务接口创建跟进任务或发送通知。
12. `Report Agent` 生成结构化分析报告，并保存到数据库和对象存储。
13. 前端通过 WebSocket 或 SSE 实时展示任务执行状态和最终结果。

### 1.6 非功能性需求

- 性能：普通业务接口 P95 响应时间小于 300ms；AI 长任务异步处理，前端实时展示进度。
- 可用性：核心服务支持多副本部署；任务状态持久化，Agent 失败后可重试或转人工。
- 扩展性：支持新增 Agent、工具、模型供应商、知识库类型和业务插件。
- 安全性：基于 RBAC 与租户隔离控制数据访问；敏感字段脱敏；模型调用日志可审计。
- 可观测性：所有服务接入 OpenTelemetry，支持日志、指标、链路追踪和 Agent 执行轨迹分析。
- 可测试性：业务逻辑、Agent 节点、工具调用、RAG 检索和端到端流程均应具备自动化测试入口。

## 2. 系统架构

### 2.1 总体架构

系统采用前后端分离、Java + Python 双后端、事件驱动和云原生部署架构。

- 前端工作台负责用户交互、任务状态展示、知识库管理和报告可视化。
- API Gateway 作为统一入口，负责路由、鉴权、限流、跨域和灰度策略。
- Java 后端负责用户权限、工单、审批、任务、通知、审计和 Spring AI 轻量能力。
- Python 后端负责 LangChain、LangGraph、RAG、多 Agent 编排、数据分析和工具调用。
- 消息队列用于 AI 长任务、文档处理、通知发送和跨服务事件解耦。
- Kubernetes 负责服务编排、弹性伸缩、配置管理和部署治理。

### 2.2 服务划分

| 服务 | 技术 | 职责 | 部署方式 |
|---|---|---|---|
| `web-console` | React / Vue + TypeScript | 运营工作台、任务看板、报告预览、知识库管理 | K8s Deployment + Nginx |
| `api-gateway` | Spring Cloud Gateway | 统一入口、鉴权、限流、路由、服务聚合 | K8s Deployment |
| `identity-service` | Spring Boot | 用户、角色、权限、租户、JWT、审计 | K8s Deployment |
| `ticket-service` | Spring Boot | 工单、客户、SLA、任务管理、处理记录 | K8s Deployment |
| `workflow-service` | Spring Boot + Spring AI | 审批流、人工确认、业务动作、Java 侧 AI 能力 | K8s Deployment |
| `notification-service` | Spring Boot | 邮件、站内信、Webhook、消息通知 | K8s Deployment |
| `agent-service` | FastAPI + LangGraph | 多 Agent 编排、工作流状态机、任务执行状态 | K8s Deployment |
| `rag-service` | FastAPI + LangChain | 文档解析、切片、Embedding、向量检索、引用溯源 | K8s Deployment |
| `analytics-worker` | Python Worker | 异步数据分析、报表生成、模型评测、批处理任务 | K8s Deployment / Job |

### 2.3 Agent 工作流架构

核心 Agent 设计如下：

| Agent | 职责 | 输入 | 输出 |
|---|---|---|---|
| `Planner Agent` | 解析用户意图，拆解任务计划 | 用户自然语言任务、用户上下文 | 任务步骤、所需工具、执行顺序 |
| `Retriever Agent` | 检索知识库、历史工单和业务数据 | 查询意图、过滤条件 | 相关文档、数据片段、引用来源 |
| `Data Analyst Agent` | 分析趋势、异常、分类分布和热点问题 | 工单数据、客户数据、历史指标 | 分析结论、图表数据、异常点 |
| `Risk Agent` | 识别高风险客户、工单和 SLA 风险 | 分析结果、业务规则、客户画像 | 风险等级、风险原因、处置建议 |
| `Supervisor Agent` | 校验输出质量和动作风险 | 各 Agent 输出、置信度、引用来源 | 是否通过、是否需要人工确认 |
| `Action Agent` | 调用业务系统执行动作 | 审批结果、动作参数 | 派单结果、通知结果、任务 ID |
| `Report Agent` | 生成结构化报告 | 执行过程、分析结论、引用来源 | Markdown / HTML / PDF 报告 |

### 2.4 通信与数据流

| 场景 | 通信方式 | 说明 |
|---|---|---|
| 普通业务查询 | HTTP / REST | 前端经网关访问 Java 业务服务 |
| AI 长任务提交 | REST + MQ | Java 建档后投递消息，Python Agent 异步处理 |
| 执行过程推送 | SSE / WebSocket | 前端订阅任务状态与 Agent 步骤输出 |
| 工具调用 | 内部 REST / gRPC 可选 | Python Agent 调用 Java 后端提供的业务工具 |
| 文档处理 | 异步队列 | 文档上传后异步解析、切片、Embedding 和索引 |
| 日志与追踪 | OpenTelemetry | 服务链路、模型调用、工具调用和异常统一追踪 |

### 2.5 中间件架构

| 中间件 | 用途 |
|---|---|
| Redis | 缓存、限流、任务进度、分布式锁、热点数据 |
| Kafka / RabbitMQ | AI 长任务、文档处理、通知事件、服务解耦 |
| PostgreSQL / MySQL | 用户、权限、工单、任务、审批、审计等核心业务数据 |
| Milvus / pgvector | 文档切片 Embedding、语义检索、RAG 召回 |
| MinIO | 原始文档、附件、生成报告、导出文件 |
| Elasticsearch / OpenSearch | 工单全文检索、文档检索、日志检索 |
| Prometheus + Grafana | 指标采集、监控大盘、告警 |
| Jaeger + OpenTelemetry | 分布式链路追踪、Agent 执行轨迹分析 |

### 2.6 Kubernetes 部署架构

- 每个微服务独立构建 Docker 镜像。
- 使用 Helm 管理部署模板、服务配置和环境差异。
- 网关、Java 服务、Python 服务、Worker 服务分别配置 `Deployment`、`Service`、`ConfigMap` 和 `Secret`。
- Agent 服务和 Worker 可根据队列长度、CPU、内存或自定义指标进行 HPA 扩缩容。
- 开发环境可使用 `docker-compose` 快速启动数据库、中间件和服务。
- 生产环境建议使用托管数据库或 K8s Operator 管理关键有状态组件。
- 所有服务统一接入日志、指标和链路追踪。

### 2.7 安全架构

- 认证：基于 JWT 或 OAuth2/OIDC。
- 授权：基于 RBAC，按租户、角色、资源和动作控制权限。
- 数据隔离：所有核心业务表包含 `tenant_id`。
- 敏感数据：客户手机号、邮箱、证件号等字段需要脱敏展示。
- AI 安全：Prompt、模型输入、工具调用参数和模型输出需要审计。
- 工具调用安全：高风险工具必须经过审批或规则校验。
- 操作审计：用户操作、Agent 动作、审批记录和模型调用均需留痕。

## 3. 代码结构

建议采用 Monorepo 管理，便于统一版本、统一 CI/CD、统一部署脚本和端到端测试。

```text
agentops-hub/
  apps/
    web-console/                 # 前端运营工作台

  services/
    identity-service/            # 用户、租户、角色、权限
```

### 4.3 主要表设计

#### tenant

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | bigint / uuid | 主键 |
| `name` | varchar | 租户名称 |
| `code` | varchar | 租户编码 |

### 4.4 关键索引建议

- 所有核心业务表增加 `tenant_id` 组合索引，支撑多租户隔离查询。
- `sys_user` 建议建立 `tenant_id + username` 唯一索引。
- `ticket` 建议建立 `tenant_id + status`、`tenant_id + priority`、`customer_id`、`created_at`、`sla_deadline` 索引。
- `ai_task` 建议建立 `tenant_id + status`、`created_by`、`created_at`、`task_type` 索引。
- `agent_step` 建议建立 `task_id`、`run_id`、`agent_name`、`status` 索引。
- `document_chunk` 在关系库保存元数据，向量字段放入 pgvector 或 Milvus，并建立向量索引。
- `tool_call_log` 建议建立 `task_id`、`step_id`、`tool_name`、`created_at` 索引。
- `audit_log` 和 `model_call_log` 可按时间分区，避免长期运行后日志表过大。

### 4.5 数据一致性与事务边界

- Java 业务服务内部使用本地事务保障工单、审批、任务状态等核心数据一致。
- Java 与 Python 跨服务动作采用事件驱动和状态机保证最终一致性。
- Agent 每个关键步骤都写入 `agent_step`，避免长任务失败后无法恢复。
- 业务动作通过 `action_command` 表记录幂等键，防止 Agent 重试导致重复派单或重复通知。
- 文档上传先写入 `document`，再异步生成 `document_chunk` 和向量索引；失败时记录处理状态并允许重试。

- Java 与 Python 服务之间的 REST / MQ 通信。
- 文档上传到解析、切片、Embedding、检索的完整链路。

### 5.3 系统测试

覆盖端到端业务场景：

1. 用户登录。
2. 上传知识库文档。
3. 导入或创建客户工单。
4. 发起投诉分析 Agent 任务。
5. Agent 检索知识库和工单数据。
6. 生成风险判断和处理建议。
7. 触发人工审批。
8. 审批通过后创建跟进任务。
9. 生成报告并展示在前端。

### 5.4 压力测试

- 使用 JMeter 测试 Java 普通业务接口并发能力。
- 使用 Locust 测试 AI 任务提交、状态查询和报告获取能力。
- 模拟大量文档上传，测试 RAG 索引构建吞吐。
- 模拟高并发 WebSocket / SSE 连接，测试任务状态推送能力。
- 模拟队列积压，观察 Worker 扩缩容效果。

### 5.5 AI 质量评测

- RAG 命中率：检索结果是否包含正确答案来源。
- 引用准确率：报告中的引用是否可追溯到真实文档。
- 幻觉率：模型是否生成无来源或错误结论。
- 工具调用成功率：Agent 调用业务工具是否成功。
- 工作流完成率：复杂任务是否能从开始执行到最终报告生成。
- 人工介入率：哪些任务因为低置信度或高风险进入人工审批。
